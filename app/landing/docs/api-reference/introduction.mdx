## Base URL

```
https://your-domain.com/api
```

For local development: `http://localhost:3000/api`

## Authentication

Authenticate using one of the following headers:

| Header | Use case |
|--------|----------|
| `x-api-key` | Server-side or dashboard API key |
| `x-session-token` | User session (e.g. dashboard logged-in user) |

All endpoints that require auth return `400` with `{ "error": "API key or session token is required" }` when missing.

<Note>
  Get your API key from your Stellar Tools dashboard (Settings → API Keys).
</Note>

## API resources

| Resource | Description |
|----------|-------------|
| [Checkout](/api-reference/checkout) | Create and manage payment checkouts (one-time or subscription) |
| [Customers](/api-reference/customers) | Create, list, get, update, and delete customers |
| [Webhooks](/api-reference/webhooks) | Create, update, and delete webhook endpoints |
| [Subscriptions](/api-reference/subscriptions) | Create and manage subscriptions (pause, resume, cancel) |
| [Payments](/api-reference/payments) | Retrieve payment status |
| [Refunds](/api-reference/refunds) | Create refunds for payments |
| [Products](/api-reference/products) | Create products (one-time, subscription, or metered) |
| [Credits](/api-reference/credits) | Manage credit balances and usage for metered billing |
| [Payout](/api-reference/payout) | Send payouts to a Stellar address (session only) |

## Stellar discovery (SEP-1)

For wallet and client discovery, the API exposes:

- **Organization stellar.toml** — `GET https://{orgId}.stellartools.io/.well-known/stellar.toml` (or your org subdomain). Query `?mode=testnet` or `?mode=mainnet` for network. Returns TOML with `CURRENCIES`, `ACCOUNTS`, `WEB_AUTH_ENDPOINT`, and org info.
- **Platform stellar.toml** — `GET /api/.well-known/stellar.toml?mode=mainnet|testnet` returns network passphrase and `URI_REQUEST_SIGNING_KEY` for SEP-7.
- **SEP-1 org info** — `GET /api/sep1?mode=mainnet|testnet` (host-based org) returns TOML for the requesting organization.

## Response format

Success responses wrap data in a `data` field:

```json
{
  "data": { ... }
}
```

Errors return a JSON object with an `error` field and an appropriate HTTP status (4xx/5xx).
